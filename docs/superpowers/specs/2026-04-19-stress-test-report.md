## 性能总结

### 吞吐量

| 场景 | 单实例 QPS | 备注 |
|------|-----------|------|
| 单张图片 | 0.52 | 基准 |
| 8并发图片 | 1.38 | vLLM 批处理有效 |
| 16并发图片 | 1.95 | 接近极限 |
| 单个63页PDF | 0.019 | 每页0.8s |
| 2并发PDF | 0.032 | 有一定提升 |
| 4并发PDF | 0.025 | 排队处理 |
| 8并发PDF | 0.033 | 排队处理 |
| 混合负载 | - | PDF 挤占图片资源 |
| 持续20图片 | 0.91 | 稳定 |

### 可靠性

| 指标 | 值 |
|------|-----|
| 总请求数 | 63 |
| 成功数 | 63 |
| 失败数 | 0 |
| **成功率** | **100%** |

### 资源使用

| 指标 | 值 |
|------|-----|
| GPU 显存占用 | 15.7GB / 24GB (65%) |
| GPU 利用率 (空闲) | 0% |
| GPU 利用率 (满载) | 77% |

### 瓶颈分析

1. **VLM 推理是主要瓶颈**: PDF 每页串行推理，63页需 51s
2. **GPU 利用率未饱和**: 峰值 77%，还有提升空间
3. **PDF 挤占图片**: 混合场景下图片等待严重 (2s → 45s)
4. **显存余量**: 15.7/24GB，剩余 8.3GB

## 对 PaddleOCR Web UI 项目的影响

### 单实例架构

```
客户端 → PaddleOCR UI (5553) → HPS (5564, GPU 2)
```

### 后端优化建议

| 项目 | 建议 |
|------|------|
| 并发控制 | `Semaphore(4)` — HPS 支持 8 并发，留余量 |
| 图片/PDF 分队列 | 图片单独信号量，不被 PDF 阻塞 |
| 超时设置 | 图片 15s，PDF 每页 5s (63页 = 315s) |
| 文件大小限制 | 建议 100MB 上限 |
| 文件格式 | 图片: JPG/PNG/BMP/TIFF/WebP, 文档: PDF 直传 |
| Word/Excel | 后端需先转 PDF 再传 OCR 服务 |
| 大 PDF 提示 | 超过 100 页的 PDF 前端提示"预计处理时间较长" |
